1️⃣ Introduction

The Project: Satellite Imagery at INSEE

  • Goal: detect buildings on satellite imagery across French overseas territories (OTs)
  • Identify areas with the strongest changes in built-up land — constructions & demolitions
  • Priority targets: informal settlements in French OTs
  • Used to assist and optimise cartographic surveys conducted before each census

Demo: web application

[Image: pipeline of the project]

Bonjour à tous. Aujourd’hui, nous allons vous présenter l’utilisation d’Onyxia dans un contexte géospatial, en nous appuyant sur un cas d’usage concret : le projet Images Satellites de l’INSEE.

Ce projet a pour objectif de détecter les bâtiments sur des images satellites afin d’identifier les zones du territoire qui subissent les évolutions les plus marquées du bâti — qu’il s’agisse de constructions ou de destructions. L’enjeu est de surveiller les zones à fort “mouvement”, comme les bidonvilles dans les départements et régions d’outre-mer français (DROM).

Full technical stack of the project

Entire pipeline built and run on Onyxia

  • Use Case Cyclone Chido hit Mayotte in late 2024 — one of the most devastating in decades
  • Using our pipeline, we produced a full post-cyclone land cover map and identified destructions in a single night of computation

Ce projet a été entièrement développé sur Onyxia et sa pipeline est constituée exclusivement de briques Onyxia. Lors du cyclone Chido à Mayotte fin 2024, il a été possible de produire une couverture du sol post-cyclone et d’identifier les destructions en une seule nuit de calculs. Nous allons revenir étape par étape sur cette pipeline pour l’expliquer en détail.

À noter : le backend de ce projet est entièrement développé en Python.

2️⃣ Onyxia

What is Onyxia SSPCloud?

  • Browser-based data science workspace — no local installation
  • Launch Python, R environments with CPU / GPU allocated on demand
  • Built on Kubernetes → scalable, reproducible, cloud-native
  • Integrated S3 object storage — or connect to an external S3 bucket
  • Project spaces: share services & data across a team

Onyxia SSPCloud est un environnement de travail pour la data science, accessible directement depuis votre navigateur. Il permet de lancer des sessions de développement (Python, R, etc.) avec des ressources CPU et GPU allouées à la demande, sans installation locale. Cette plateforme repose sur Kubernetes, ce qui lui confère flexibilité et scalabilité.

Elle dispose d’un stockage objet S3 intégré où l’on peut stocker et accéder à ses données. Il est également possible de se connecter à un bucket S3 externe pour y récupérer des données tierces.

Il est possible de demander la création d’un “projet” Onyxia afin de partager services et données entre plusieurs utilisateurs. Dans notre cas, nous disposons d’un projet dédié au projet Images Satellites, ainsi qu’un bac à sable isolé — c’est dans cet environnement que je réaliserai mes démonstrations, afin de ne pas interférer avec les services et données en production.

Services Used in Our Project

  • VSCode / VSCode GPU
    • Browser-based IDE, central tool for development & experimentation
    • GPU variant for deep learning training (SegFormer fine-tuning)
  • MLFlow
    • Experiment tracking: metrics, hyperparameters, artefacts
    • Model Registry: promote a run to Production with one click
  • Argo Workflows
    • Kubernetes-native pipeline orchestration
    • DAG-based: each step is a container, dependencies are explicit
    • Enables parallel processing across image tiles
  • ArgoCD
    • GitOps continuous deployment
    • Watches our Git repo, auto-syncs the cluster on push
    • Used to deploy & scale our inference API

  • VSCode / VSCode GPU : environnement de développement intégré accessible depuis le navigateur. La version GPU permet d’entraîner des modèles de deep learning en tirant parti d’accélérateurs graphiques. C’est notre outil central pour le développement, le débogage et l’expérimentation.

  • ArgoCD : outil de déploiement continu basé sur GitOps. Il surveille un dépôt Git et synchronise automatiquement l’état du cluster Kubernetes avec la configuration déclarée. Nous l’utilisons pour déployer et mettre à jour notre API d’inférence.

  • Argo Workflows : moteur d’orchestration de workflows Kubernetes. Il nous permet de lancer des pipelines de calcul complexes (entraînement, inférence) de manière parallélisée et reproductible, directement sur le cluster.

4️⃣ GeoServer

What is GeoServer?

  • Open-source geospatial data server
  • Publishes raster & vector data via OGC standards:
    • WMS (Web Map Service) — rendered map tiles
    • WFS (Web Feature Service) — raw vector features
  • Layers are directly consumable by web apps

Un GeoServer est un serveur open source de données géospatiales. Il permet de publier, stocker et servir des données géographiques (rasters, vecteurs) via des protocoles standardisés comme WMS (Web Map Service) ou WFS (Web Feature Service). Il est ainsi possible de diffuser des couches cartographiques consommables directement par des applications web.

Deploying GeoServer on Onyxia

  • Not in the Onyxia service catalogue — but deployable on the underlying Kubernetes cluster
  • Requirements:
    • VSCode service with Kubernetes admin role
    • Define Deployment, Service, Ingress resources manually
  • Once deployed: accessible via a cluster Ingress URL like any other service

Le GeoServer ne fait pas partie du catalogue de services Onyxia, mais il est tout à fait possible de le déployer manuellement sur le cluster Kubernetes sous-jacent. Cela nécessite un service VSCode avec le rôle Kubernetes admin, ainsi que la maîtrise des ressources Kubernetes : Deployment, Service et Ingress. Je n’entrerai pas dans ce niveau de détail ici, mais sachez que l’opération est réalisable.

Pipeline → GeoServer Integration

  • Inference pipeline automatically pushes results to GeoServer on completion:
    • Land cover and Building change polygons → new WFS layers
  • GeoServer creates and versions the layers automatically

Notre pipeline d’inférence est directement connectée à notre GeoServer : à chaque exécution, les couvertures du sol prédites ainsi que les évolutions du bâti détectées sont automatiquement déversées dans le GeoServer, qui crée et met à jour les couches géographiques associées.

Observable Framework

  • Frontend built with Observable Framework — JS-based dataviz platform
  • Combines markdown, JavaScript, and live data in reactive notebooks
  • Consumes GeoServer layers via WMS / WFS for live map rendering

Full stack: Python backend · JavaScript, HTML, CSS frontend

Pour finir, notre application web de visualisation a été développée avec Observable Framework (anciennement Observable.js). Il s’agit d’un framework JavaScript orienté dataviz et analyse, qui permet de créer des tableaux de bord interactifs mêlant code, texte et visualisations cartographiques. Nous l’utilisons pour afficher les couches géospatiales issues de notre GeoServer et permettre à nos utilisateurs d’explorer les évolutions du bâti sur le territoire.

J’avais mentionné que notre backend était entièrement en Python — notre frontend, quant à lui, mobilise JavaScript, HTML et CSS.

Development & Deployment Workflow

  • Develop & preview locally in a VSCode (Onyxia) service with npm
  • Git push → GitHub Actions build → auto-deploy to GitHub Pages
  • Zero server to maintain, free hosting

[Demo: live application — inseefrlab.github.io/satellite-images-webapp]

Le workflow de développement est le suivant : les modifications sont effectuées dans un service VSCode Onyxia et visualisées en local grâce à npm. Une fois les changements validés, un push sur Git déclenche automatiquement le build de l’application et son déploiement sur GitHub Pages.
